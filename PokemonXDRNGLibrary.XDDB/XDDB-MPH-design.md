# XDDB（最小完全ハッシュ版）設計メモ

XDDB を「コード→seed の最小完全ハッシュ（MPH）」で保持し、hp キー列を持たない方針と、その成立根拠をまとめる。

## 1. 背景・現状

XDDB は QuickBattle の検索用インデックスで、本体は `(hpCode, seed)` の組の集合。

- レコード数: 221791
- 異なる hpCode 数: 219722（1コードあたり平均 1.0094 seed、最大 3、複数 seed は 2,040 コード）
- hpCode: 7bit フィールド 4 つ（eFirst, eSecond, pFirst, pSecond、各 0..94）を詰めた値。ソートキー。
- seed: 24bit（上位 8bit = h8 は保存せず、検索時に総当たり）。ほぼ一様乱数で圧縮は効かない。

### 既存バージョン

| Ver | ファイル | 1レコード | サイズ | ランダムアクセス二分探索 |
|---|---|---|---|---|
| V1 | XDDB.bin | hp4 + seed4（固定 8B） | 1774328 | 可 |
| V2 | XDDB2.bin | hp4 + seed3（固定 7B） | 1552537 | 可 |
| V3 | XDDB3.bin | varint(hp差分) + seed3（可変長） | 1052582 | 不可 |

「固定長＋絶対値キー」を保つ限りの下限は 52bit/件（hp 28bit + seed 24bit）= 約 1.44MB。
それ未満にするには「差分」か「可変長」が必要になり、固定オフセット参照（i 番目への直接到達・前を読まずにキーを得る）が成立しなくなる。

## 2. 方針

検索の本質は「与えられた hpCode に対応する seed を引く」ことで、テーブル中の hp は引くためのキーとしてしか使っていない。この点を利用し、hp 列を完全ハッシュ（MPH）で置き換える。

- 219722 個の異なる hpCode に対して Minimal Perfect Hash `mph(code) → slot ∈ [0, k)` を構築。
- seed を slot 順に並べた配列に格納。
- 検索: `slot = mph(queryCode)` → `seed = seeds[slot]` → 既存通り h8 を 0..255 総当たりして生成・検証。

ただし、結果の列挙順序が slot/overflow 順に変わってしまう点に注意。

### データ構造（XDDB.bin）

```
[ヘッダ]   k（コード数）, バケット数, 葉サイズ, 各バケットのキー数・ビット長
[MPH]      RecSplit の seed 列（Golomb-Rice 符号化）              ≈ 2bit/件
[seeds]    slot順に並べた 24bit seed                          221791 × 3B
[overflow] 複数seedを持つ 2040コード分の追加seedと対応構造    約12KB（§4参照）
```

## 3. 成立根拠

### (A) hp はキーであり、復元対象でも照合対象でもない
- 復元不要: hpCode はクエリ（プレイヤーの観測値 = `QuickBattleInput.HPCode`）が持参するため、テーブル側で hp を再現する必要はない。
- 照合不要: `Search` は候補 seed から `(h8<<24 | seed).GenerateQuickBattle()` を生成し、`first.Check`（pIndex / eIndex / HPCode 一致）で再検証している。照合は生成側が担う。

→ hp の役割は「seed の位置を引くインデックス」のみ。インデックスを別手段で引ければ hp 列は不要。

### (B) MPH はキーを保存しないので、コードの bit 幅と無関係に小さい
- MPH が保存するのはハッシュ関数のパラメータ（seed 列）のみであり、コードそのものは保存しない。
- 本件では RecSplit を採用する:
  1. ハッシュでコードをバケットに振り分ける。
  2. 各バケット内で二分分割の木を構築する。内部ノードでは、左の子にちょうど半数のキーが落ちる split seed を探索する。
  3. 葉（サイズ ≤ 8）では、全キーが [0, size) へ一対一に写る bijection seed を探索する。
  4. seed 列を Golomb-Rice 符号で圧縮して格納する。
- RecSplit の MPH サイズは約 2bit/件となり、入力されるコードのサイズには依存しない。本件ではおおよそ 49KB になる。

### (C) 集合に無いコードを引いても誤った結果は返らない

前提として、MPH は元の集合に無いコードが渡された場合でも何らかの slot を返してしまう。通常はその対策としてフィンガープリントを併用して照合する。
しかし本件では、後段に『候補seedに対して検算を行う処理』が入る。したがって、MPH が無関係な seed を返してもここで除外されるため、MPH側での特別の対策は不要。

（検算を通ったのであればそれは正しい解であり、偽陽性はあり得ない。）

ただし、現行実装では、存在しないコードが渡された場合、検算処理に入る前の段階で枝刈りされる。MPHを使う実装では、このようなケースに対するパフォーマンスは下がってしまう。

## 4. サイズ見積もり

| 要素 | サイズ |
|---|---|
| seed 列 24bit × 221791（下限・必須） | 665373 |
| MPH seed 列（RecSplit）~2bit/件 × 219722 | ≈ 49000 |
| 複数 seed コードの追加 seed（2069 個 × 3B） | 6207 |
| overflow を持つ slot を引く構造 | ≈ 6000（ソート済み slot リスト方式） |
| 合計 | ≈ 720〜730KB |

- overflow を引く構造は、対象 slot をソートして持ち、クエリ時に二分探索する方式で約 6KB。bitmap 方式（1bit × 219722 ≈ 27KB）も可能だが効率は劣る。
- 参考: V2 1.55MB

## 5. メモリ・I/O 特性

- 現行 V1〜V3 は読み込み時に全データを 2 本の `List<uint>`（`hpData`/`seedData`、各 221,791 件）へ展開して常駐する（`XDDBClient.LoadV1/2/3`）。
  - 使用するメモリは約 1.7MB
  - 検索は二分探索 O(log n)
- V1/V2 では、データをメモリに常駐させず、クエリごとにファイルを直接読む実装も可能。
  - 使用するメモリはファイルハンドルのみで、無視できると考えて良い
  - 検索は二分探索で ~18回（`log₂ 221791 ≒ 18`）の seek とレコードの読み出し（ランダムI/O）。加えて、該当後に同一コードの隣接（≤3）を読む
- MPH 版では、メモリに全データを常駐させる実装と、MPH ブロックのみを常駐させる実装が考えられる
  - 全データを常駐させる場合
    - 使用するメモリは MPH ブロック 49KB + seeds 665KB + overflow ~6KB ≈ 720KB
    - 検索はハッシュ評価 O(1)
  - MPH ブロックのみを常駐させる場合
    - 使用するメモリは MPH ブロックの約 49KB
    - 検索はハッシュ評価のあと、1回の seek と読み出し（ランダムI/O）。加えて、overflow のコードのみ、追加で数件読む
- ただし、検索処理については、いずれの実装でもクエリの実コストは`h8`の総当たりが支配的で、上記の差は実用上ほぼ表に出ない

## 6. 注意点

- ハッシュ関数（Mix）と Rice パラメータの導出ロジックは、MPH 構築側と参照側で同じものを使う
- 構築したMPHの `d[]` が衝突しないかどうか実際に検証する必要がある
  - 生成後に全 219722 コードを評価式へ通し、slot が 0..k-1 の一対一になっていることを確認する

## 7. ライブラリ構成

### 動機

XDDB のデータ総量は約 720KB あり、ライブラリの同梱データとしては大きく、このサイズのデータが埋め込みリソースとして提供されることが利用者の不都合になる場合が考えられる。
利用者がアプリケーションの要件に応じて、データの持ち方を選べるようにしたい。

### データの内訳

| 部位 | 内容 | サイズ |
|---|---|---|
| index部 | MPH ブロック（RecSplit の seed 列）≈ 49KB + overflow ≈ 12KB | ≈ 61KB |
| seeds部 | u24 × 221,791 件 | ≈ 659KB |
| 合計 | | ≈ 720KB |

index部はサイズが十分小さいためコアパッケージに含めるようにし、seeds部のみを切り出す。

### index部の持ち方

MPH ブロックのパラメータ（バケットサイズ、ビットオフセット、seed ビット列）や overflow の slot リスト等は、実行時に使う形式（`int[]`、`byte[]` 等）でソースコードに直接定義する。当該ソースコードは `XddbBuilder` で自動生成する。

### パッケージ分割

NuGet パッケージを`XDDB`と`XDDB.Data`の2つに分割する。

- `XDDB`は検索ロジックと index部を持ち、seedsの供給元を引数に取るAPIを公開する。
- `XDDB.Data`はseedsを埋め込みリソースとして持ち、`Stream` 等を返すAPIを通じて公開する。

### バージョン検証

index 部と seeds 部は MPH の slot 割り当てによって対になっているため、対応が崩れると誤った seed を返してしまう。
これを防ぐために、双方にバージョン値を埋め込み、データベースを開く際に一致を検証する。

### XDDB.bin の配布

利用者が外部ファイルとして使う場合のために、seeds部をファイル出力する処理を `XddbBuilder` に含める。
